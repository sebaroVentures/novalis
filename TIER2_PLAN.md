# Novalis Tier-2 Roadmap

## Sequencing Rationale

Five XL/L bets. They are **not** independent — three of them write to the same load-bearing seam (the `pendingFlush` registry + flush-before-switch discipline in `vaultStore.ts:31,567`), and two of them depend on a third for a shared component. Build order is therefore dictated by who owns the flush registry and who consumes whom, not by raw value.

The single most important early decision: **Editor Tabs → Splits is the workspace backbone and must land its Phase 1a first.** Phase 1a converts the module-level singleton `pendingFlush` into a `Map<paneId, FlushFn>` and `saveState` into a per-path map. Every other bet that owns a live editor instance — Canvas (registers into "the SAME registry, don't fork it"), Transclusion (nested editors must *not* fire the flush path), Git sync (flush-before-sync) — assumes a flush registry that can hold more than one editor. If those bets land before Tabs 1a, they each fork their own assumptions about a single-active editor and Tabs 1a then has to retrofit them. Tabs 1a is invisible (zero UI, zero data-loss risk), small, and depends on nothing. It is the keystone.

**Best FIRST bet to build: Editor Tabs → Splits, Phase 1a.** Reasons: (1) it unblocks the flush-registry assumption every other live-editor bet relies on; (2) it ships invisibly so it can't regress UX; (3) it depends on nothing; (4) the spec itself calls it "the workspace BACKBONE … those bets must not also fork their own pane model."

After 1a, the rest parallelizes against a stable seam. Transclusion is the second priority because **Canvas Phase 3 explicitly depends on Transclusion's read-only `NoteEmbed` component**, and Transclusion's Phase 1a is a self-contained Rust command with no editor coupling. Properties and Graph are both backend-first, low-coupling, and can slot in opportunistically. Git sync is heaviest/riskiest and should start with its Phase 2 spike gated behind a maintainer decision.

## Bet 1: Editor Tabs → Split Panes (workspace backbone) — XL

**Architecture.** One editor instance **per visible pane**, never per tab. `NovalisEditor` is uncontrolled — `value` is consumed once at `useEditor({ content: value })` (`NovalisEditor.tsx:228`); switching docs requires a full React remount, which is exactly what today's `key={`${activePath}:${activeNoteVersion}`}` (`EditorPane.tsx:573`) already does. Background tabs are inert descriptors `{ path, scrollTop?, dirty? }`, not live editors — keeping live TipTap instances per background tab would multiply ProseMirror state, decoration plugins (WikiLink/Math/Mermaid/Find) and the per-keystroke serialize timer N-fold for zero benefit. Workspace state (panes, tabs, focused pane) moves into a new `uiStore.workspace` (device-local, localStorage keyed by vaultPath, mirroring `sidebarPrefs`), while `vaultStore` stays the content cache + IPC owner. The load-bearing change: the single module-level `pendingFlush` (`vaultStore.ts:31`) becomes `Map<paneId, FlushFn>` and `saveState` becomes `Map<path, SaveState>` — the write path (`saveNote(path, content)` idempotent via `lastRequest`, `vaultStore.ts:25,541`) is already pane-safe; only the registry and status surfacing are single-active. Rejected: iframe/portal-per-tab (breaks shared decoration registries), making NovalisEditor controlled (fights TipTap, risks round-trip regressions), and a `panes` array in vaultStore (conflates content cache with workspace UI).

**Top risks.** (1) **Same-note remount loop** (highest risk): a save bumping noteVersion on the typing pane remounts mid-edit, losing cursor — gate strictly on `focusedPaneId`, mirror only NON-focused panes; needs careful manual alternating-edit testing. (2) Per-keystroke serialize timers/decoration plugins multiply per live editor — mitigated by max 2-4 visible panes, inert background tabs. (3) Wide blast radius retargeting every open-note call site — single `openInWorkspace` choke point, `openNote` becomes its internal content-loader. (4) `vaultStore.history/historyIndex` (164-165) is a global single stack; v1 ties it to the focused pane (acceptable, per-pane history is a known follow-up, not silently "fixed"). (5) EditorPane's local-state effects keyed on activePath (211-246) — verify each effect's deps when threading paneId or risk stale reading-mode/find state.

## Bet 2: Transclusion / Embeds — `![[Note]]`, `![[Note#Heading]]`, `![[image]]` — L

**Architecture.** Block-level `Decoration.widget` from a ProseMirror plugin (the WikiLink/Math/Callout house style) — **NOT** a custom ProseMirror node, **NOT** a tiptap-markdown extension. The load-bearing invariant is that everything renders off plain text via decorations so tiptap-markdown's serializer never learns a new node (WikiLink.ts:1-4, Math.ts:91-93, Callout.ts:10-12); a custom node would force a parse+serialize rule whose bugs silently rewrite the user's file. The widget anchors at the start of the text-block containing a standalone `![[...]]`, rendered `side:-1` + `contentEditable=false`; the source text is hidden via inline decoration `class:"nv-embed-src"` (`display:none`, mirroring `.nv-math-src` editor.css:200) unless the cursor is inside the run (reveal-for-edit, copy Math.ts:60). Note embeds render through a nested `<NovalisEditor editable={false}>` mounted via `createRoot` so math/mermaid/callouts/nested wikilinks render exactly as the source, keyed by `target`+content-hash. New host callback `onResolveEmbed(ref) => Promise<EmbedResult>`; EditorPane classifies images by extension (reuse `resolveImageSrc`) and resolves notes via a new `api.resolveEmbed(target)` core command, cached per-target (`Map<string,EmbedResult>`, like Math.ts:14 `htmlCache`). Recursion guarded by an `embedDepth` prop (MAX_EMBED_DEPTH≈3 → plain "open note" chip beyond). Rejected: custom Node + tiptap-markdown rule (breaks round-trip), pure-frontend quickSearch+getNote (duplicates alias resolution in TS, forces JS section-slicing).

## Bet 3: Typed Properties Panel (structured frontmatter editor) — L

**Architecture.** Do **NOT** add custom-property struct fields to `NoteFrontmatter`, do **NOT** change `serialize_frontmatter`. The `extra: serde_json::Value` passthrough (note.rs:32-34) already preserves unknown keys; the whole feature is (a) exposing that map and (b) editing it through a narrow command surface. Value model = `serde_json::Value` limited to String/Number/Bool/Array<String>/Null; serde_yaml maps these cleanly (a date string `2026-06-10` round-trips as a bare scalar, not re-quoted) so "date" is a UI affordance over a JSON String, not a wire type. Key ordering is free: `serde_json::Value::Object` is a BTreeMap → serde_yaml emits extra keys alphabetically and round-trip-stably (verified); known struct fields serialize first in declaration order, then extra alphabetically. The only diff-noise event is a one-time alphabetical reorder when first rewriting a hand-ordered legacy note — acceptable. Do **NOT** enable serde_json `preserve_order` (makes ordering edit-history-dependent → MORE sync noise). Expose via a sibling computed field `properties: Vec<NotePropertyEntry>` on `Note` (derived from `extra` in read_note), with `PropertyValue` as a `#[serde(tag="kind",content="value")]` enum — keeps the wire type small/closed while on-disk YAML stays open. Command surface on `notes`: `set_property`/`remove_property`/`rename_property`, all read-parse-mutate-serialize-write through the `update_meta` path (notes/mod.rs:95) so they get reindex + flush-before-switch; reserved-key guard rejects title/tags/aliases/created/modified/pinned with `BadRequest`.

## Bet 5: Git-backed sync (local-first multi-device) — XL

**Architecture.** Library: **gix (gitoxide)**, NOT git2/libgit2, NOT shelling to `git`. gix is pure-Rust (no libgit2/OpenSSL C dep → clean cross-compile for the existing macOS/linux/windows targets and the planned mobile target), memory-safe APIs for init/open/status/diff/commit/fetch/push/merge, and keeps the "everything bundled, no external runtime deps" posture (rusqlite `features=["bundled"]`). **Caveat front-loaded**: gix's high-level *merge* and *push* are its thinnest surfaces — the phasing front-loads rock-solid commit/log/fetch and defers merge/push to Phase 2/3, where a timeboxed spike against a real remote decides whether to fall back to **git2 behind a `sync-remote-git2` feature flag for push/3-way-merge ONLY** (not a wholesale swap). Auto-merge = git's own 3-way line merge; true conflicts surface through the existing ConflictModal (adapted to 3-way) instead of raw `<<<<<<<` markers. (3) **Phase 3 hard dependency on Transclusion's NoteEmbed** — gated; the fallback (title+excerpt) ships Phase 3 standalone if Transclusion isn't ready, but the preferred path needs it. (4) react-flow default styling bleed — explicitly token-style nodes/edges/minimap/controls/background. (5) Vite `canvas` chunk split — verify `@xyflow/react` lands in the `canvas` chunk via build output.

## Recommended Order — First ~5 PRs Across All Bets

1. **Tabs/Splits Phase 1a** — the keystone. Converts `pendingFlush` → `Map<paneId,FlushFn>` and `saveState` → per-path map. Invisible, zero data-loss risk, depends on nothing, unblocks every live-editor bet. **Build first, no exceptions.**
2. **Transclusion Phase 1a** (`resolve_embed` core command) — self-contained Rust + IPC, no editor coupling, fully testable in isolation. Unblocks Transclusion 1b *and* Canvas Phase 3's NoteEmbed. Can be built in parallel with PR1 (different files: core/notes vs frontend stores).
3. **Tabs/Splits Phase 1b** (tab strip) — the headline user-visible feature, lands on the stable 1a seam. Retargets all open-note call sites through the single `openInWorkspace` choke point — do this before other bets add their own open-note paths (Canvas, Graph nav).
4. **Graph Phase 1** (`full_graph` query) OR **Properties Phase 1** (backend) — both are independent backend-first slices with their own Rust tests and bindings, zero coupling to the editor seam. Pick Graph 1 if the maintainer wants visible graph value soonest; pick Properties 1 if structured metadata is the priority. Either is safe in parallel with PR3.
5. **Transclusion Phase 1b** (Embed extension) — delivers inline embeds and produces the `NoteEmbed`-shaped component that Canvas Phase 3 reuses. Lands after Tabs 1b so it inherits the per-pane editor parameterization cleanly rather than fighting a mid-flight refactor.

This front-loads the two keystones (flush registry, resolve_embed), ships the headline tabs feature early, and keeps two independent backend bets available to fill parallel capacity without touching the editor seam.

## Architecturally Heaviest Decisions — Need Maintainer Sign-Off BEFORE Building

1. **Git library: gix vs git2-behind-a-feature-flag (Git Sync).** gix is the chosen default (pure-Rust, clean cross-compile incl. mobile, no libgit2/OpenSSL C dep), but its high-level **push and 3-way merge are its thinnest surfaces** at current versions. The plan front-loads solid commit/log/fetch (Phase 1) and gates Phase 2 behind a **timeboxed spike against a real HTTPS remote** that decides whether to add `git2` behind a `sync-remote-git2` feature for push/merge ONLY. Sign-off needed: (a) accept gix as default, (b) accept the git2-fallback escape hatch, (c) approve the Phase-2 spike as a hard gate before committing the remote phase. This is the single heaviest/riskiest bet.

2. **One-editor-per-pane vs doc-swap, and moving "the open note" out of vaultStore (Tabs/Splits).** The chosen cut — one live editor per *visible pane* (not per tab), background tabs as inert descriptors, workspace state in `uiStore.workspace`, and `pendingFlush` → `Map<paneId,FlushFn>` — is the backbone three other bets assume. The same-note-in-two-panes mirror-on-save rule (remount only NON-focused panes) is the highest-risk single piece in the whole roadmap. Sign-off needed because reversing it later (e.g. to keep live editors per tab, or to keep state in vaultStore) invalidates Canvas's flush-registry sharing and the per-pane assumptions baked into every later multi-note feature.

3. **Canvas engine: @xyflow/react (Canvas bet).** ~150KB lazy chunk, near-1:1 data-model match to JSON Canvas, MIT — chosen over tldraw (owns `.tldr`, fights the flat JSON schema, multi-MB) and custom (months of pan/zoom/hit-testing reinvention). Sign-off needed before building because it's a permanent dependency and document-format commitment; the entire CanvasView and the canvasSchema adapter are built around its node/edge shape.

Secondary decisions worth a nod but lower blast radius: **d3-force vs alternatives + the SVG/Canvas 500-node threshold** (Graph — reversible, isolated to GraphView), and **PropertyValue wire enum: f64 vs raw `serde_json::Number` for int/float fidelity** (Properties — verify the serde_yaml round-trip before committing the enum shape; cheap to get right early, annoying to change after bindings ship).
